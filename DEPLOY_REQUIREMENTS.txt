==============================================================================
 BCM (Buoyant Cluster Management for bitrix-env)
 ТРЕБОВАНИЯ К РАЗВЁРТЫВАНИЮ: машины, характеристики, IP-адреса, сетевые порты
==============================================================================
Версия документа соответствует архитектуре BCM 1.0.2 (bitrix-env 9, Galera/PXC 8.4,
ProxySQL, MinIO, HAProxy+Keepalived). Все цифры — МИНИМУМ для отказоустойчивой
работы; для продакшена ресурсы масштабировать под нагрузку и размер БД/файлов.

ОС всех узлов: Oracle Linux 9 (x86_64)


------------------------------------------------------------------------------
1. СКОЛЬКО МАШИН НУЖНО
------------------------------------------------------------------------------
Кластер состоит из 4 слоёв. Минимальная отказоустойчивая конфигурация:

  Слой  Роль   Мин. кол-во   Назначение
  ----  -----  -----------   --------------------------------------------------
  LB    lb         2         HAProxy (вход, TLS-терминация) + Keepalived (VIP)
  WEB   web        2         bitrix-env (nginx+apache+php) + ProxySQL + Redis
                             (сессии/push/кэш) + push-server + lsyncd + Keepalived.
                             «Мозг» кластера (полное меню BCM).
  PXC   pxc        3         Percona XtraDB Cluster 8.4 (Galera, синхронная
                             репликация). ОБЯЗАТЕЛЬНО НЕЧЁТНОЕ число (кворум) — 3,5,7…
  S3    s3         2         MinIO (Site Replication, active-active) — /upload и бэкапы

  ИТОГО МИНИМУМ: 2 + 2 + 3 + 2 = 9 машин (физических или ВМ).

Инвариант масштабирования (проверяется BCM при удалении узла):
  lb >= 2,  web >= 2,  s3 >= 2,  pxc >= 3 И нечётное.


------------------------------------------------------------------------------
2. МИНИМАЛЬНЫЕ ХАРАКТЕРИСТИКИ (на одну машину слоя)
------------------------------------------------------------------------------
  Слой   vCPU   RAM (мин / рекоменд.)   Диск                Примечания
  ----   ----   ---------------------   -----------------   ----------------------
  LB      2      1 ГБ  /  2 ГБ          40 ГБ               |Лёгкий (proxy+vrrp)
  WEB     4      4 ГБ  /  8 ГБ          мин. 40 ГБ          |8 ГБ если на ноде ещё
                                                            |и Transformer/LibreOffice.
                                                            |/upload вынесен в S3.
  PXC     4      4 ГБ  /  8+ ГБ         2–3 × размера БД,   |SSD минимум, желательно NVMe (Galera/IO).
                                        мин. 40 ГБ          |RAM ~ под innodb_buffer_pool.
  S3      4      2 ГБ  /  4 ГБ          = нужная ёмкость    |Лучше отдельный диск под
                                        хранения, мин 40 ГБ |данные MinIO.
                                        для системного диска|

Суммарно для минимума (9 машин): ~27 ГБ RAM в сумме (1+2+4+4+4+4+4+2+2).
Сеть: все узлы в ОДНОМ L2-сегменте/подсети (требование VRRP/Keepalived для
плавающих VIP, а также синхронной репликации Galera и redis).


------------------------------------------------------------------------------
3. СКОЛЬКО IP-АДРЕСОВ НУЖНО
------------------------------------------------------------------------------
А) Постоянные IP узлов (по одному на машину): 9 шт.
     lb01, lb02, web01, web02, db01, db02, db03, s3-01, s3-02

Б) Плавающие VIP (Keepalived), все в той же подсети:
     1) Кластерный VIP (на LB) ...... вход HTTP/HTTPS (80/443) + S3 API (9000).
                                       Сюда же резолвится домен портала и api_host S3.
     2) VIP redis-сессий (на web) .... HA хранилища сессий (порт 6380)
     3) VIP push-redis  (на web) ..... HA каналов Push&Pull (порт 6381)
     4) VIP cache-redis (на web) ..... HA кэша Bitrix (порт 6382)
     -------------------------------------------------------------------
     ИТОГО обязательных VIP: 4

ВСЕГО IP: = 9 (узлы) + 4 (VIP) = 13 постоянных IP адресов


Публичный DNS: домен портала (и, при публичной отдаче файлов, *.<vhost_domain> для
S3) должны указывать на кластерный VIP (п. Б-1). Внутри web-нод домен портала и
S3-vhost резолвятся в /etc/hosts (self-check'и Bitrix и серверная S3-интеграция).


------------------------------------------------------------------------------
4. СЕТЕВЫЕ ПОРТЫ: КТО → КУДА
------------------------------------------------------------------------------
Обозначения: tcp если не указано иное. «proto 112» = VRRP (Keepalived).
Источник истины — install.sh::configure_firewall_for_node + _bcm_redis_firewall.

--- 4.1. Клиенты (Интернет/LAN) → Кластерный VIP (LB) -------------------------
   80/tcp     HTTP  (редиректит на HTTPS, и ACME /.well-known/acme-challenge)
   443/tcp    HTTPS (весь портал; WebSocket/push идёт тем же 443 — tunnel)
   9000/tcp   S3 API (если доступ к MinIO/хранилищу нужен клиентам; через HAProxy tcp)

--- 4.2. LB  ↔  LB -----------------------------------------------------------
   proto 112  VRRP — выборы держателя кластерного VIP
   8402/tcp   ACME HTTP-01 ответчик: сосед-LB → текущий держатель VIP при выпуске
              Let's Encrypt (живёт секунды; наружу отдаёт только challenge-токен)

--- 4.3. LB  →  WEB ----------------------------------------------------------
   80/tcp     HAProxy → nginx web-нод (публичный + админский бэкенды, health-check)

--- 4.4. LB  →  S3 -----------------------------------------------------------
   9000/tcp   HAProxy (mode tcp) → MinIO API

--- 4.5. WEB  →  PXC ---------------------------------------------------------
   3306/tcp   ProxySQL (на web) → MySQL/Galera бэкенды PXC

--- 4.6. WEB  →  S3 ----------------------------------------------------------
   9000/tcp   MinIO API: модуль «Облачные хранилища» (/upload) и бэкапы (mc).
              Идёт на кластерный VIP:9000 (vhost-style) и/или прямо на s3-узлы:9000.

--- 4.7. WEB  ↔  WEB ---------------------------------------------------------
   proto 112  VRRP — VIP redis-сессий/push/cache (+ cron-роль, +transformer)
   6380/tcp   redis-сессии   — только между web-пирами (rich-rule source=web)
   6381/tcp   redis-push     — только между web-пирами
   6382/tcp   redis-cache    — только между web-пирами
   6032-6033  ProxySQL admin (6032) и proxy (6033) — управление/доступ между web
   8010-8015  push-server (sub) — внутренние порты push (nginx проксирует локально)
   5672/tcp   RabbitMQ — ТОЛЬКО при Transformer (web → VIP:5672 на держателе)
   22/tcp     SSH — lsyncd (web01→web02 синхронизация кода) и операции BCM

--- 4.8. PXC  ↔  PXC (Galera) ------------------------------------------------
   3306/tcp        MySQL (клиентские/служебные соединения)
   4567/tcp+udp    Galera group communication (gcomm/репликация)
   4568/tcp        IST — инкрементальная передача состояния
   4444/tcp        SST — полная передача снапшота при джойне
   22/tcp          SSH — авто-восстановление PXC (pxc-autorecover) и BCM

--- 4.9. S3  ↔  S3 (MinIO) ---------------------------------------------------
   9000/tcp   S3 API — Site Replication между сайтами (active-active)
   9001/tcp   MinIO Console (веб-консоль администрирования)

--- 4.10. Управляющая машина (откуда запускается install.sh) → ВСЕ узлы ------
   22/tcp     SSH — раскатка и управление (ключ /etc/bitrix-cluster/cluster_id_rsa;
              первичная раскатка ключа — по паролю root через sshpass)

--- 4.11. Узел  ↔  узел (служебное, между всеми ролями) ----------------------
   22/tcp     SSH — BCM выполняет межузловые операции по ключу cluster_id_rsa
              (статусы, failover, бэкапы, confedit, репликация Transformer и т.д.)

--- 4.12. Администратор → узлы (по необходимости) ----------------------------
   9001/tcp   → S3-узлы: веб-консоль MinIO
   (для отладки можно ходить на узлы по 22/tcp)


------------------------------------------------------------------------------
5. СВОДКА ОТКРЫТЫХ ПОРТОВ ПО РОЛЯМ (что слушает узел; firewalld)
------------------------------------------------------------------------------
  LB  : 80, 443, 9000, 6033, 8402 (tcp) + VRRP(proto112)
  WEB : 80, 443, 6032-6033, 8010-8015 (tcp) + VRRP(proto112)
        + 6380/6381/6382 (tcp, rich-rule только до web-пиров)
        + 5672 (tcp, при Transformer)
  PXC : 3306 (mysql), 4567(tcp+udp), 4568, 4444 (tcp)
  S3  : 9000, 9001 (tcp)
  ВСЕ : 22/tcp (SSH между узлами и с управляющей машины)

ПРИМЕЧАНИЯ
  • path_to_publish push и ProxySQL 127.0.0.1:3306 — локальные (loopback), наружу
    не нужны.
  • Локальный mysqld на web ВЫКЛЮЧЕН (3306 на web держит ProxySQL для self-check'ов
    bitrix-env); БД живёт только на слое PXC.
  • Весь клиентский TLS терминируется на LB (HAProxy); бэкенды web — по HTTP :80.
