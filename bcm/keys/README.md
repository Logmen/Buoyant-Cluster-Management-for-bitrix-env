# Доверенные публичные ключи релизов BCM

Файлы `*.asc` / `*.gpg` в этом каталоге — **публичные** GPG-ключи, которыми
подписываются релизы. Они раскатываются на ноды в `/opt/bcm/keys/`, и команда
`bcm --update` проверяет ими подпись скачанного релиза **перед** раскаткой
(verify-the-source). Без доверенного ключа здесь `bcm --update` откажет
(fail-closed), если только не задано `BCM_ALLOW_UNSIGNED=1` (аварийный обход).

> Сюда кладётся ТОЛЬКО публичная часть. Приватный ключ хранится у мейнтейнера и
> **никогда** не коммитится и не попадает на GitHub.

## Бутстрап (один раз, мейнтейнер)

1. Создать ключ подписи (если ещё нет):
   ```bash
   gpg --full-generate-key          # тип: ECC (ed25519) или RSA 4096; срок по вкусу
   gpg --list-secret-keys --keyid-format=long
   ```
2. Экспортировать ПУБЛИЧНУЮ часть в этот каталог и закоммитить:
   ```bash
   gpg --armor --export <KEY_ID_или_FPR> > bcm/keys/bcm-release-pub.asc
   git add bcm/keys/bcm-release-pub.asc && git commit -m "keys: add release signing public key"
   ```
3. (Опц.) Указать ключ для подписи релизов, если их несколько:
   ```bash
   export BCM_SIGNING_KEY=<KEY_ID_или_FPR>      # либо git config user.signingkey <...>
   ```

После этого `scripts/release.sh X.Y.Z` подпишет tarball этим ключом и проверит,
что соответствующий публичный ключ лежит здесь (иначе откажется публиковать).

## Ротация ключа

Добавьте новый публичный ключ рядом (старый не удаляйте сразу — тогда
верификация примет подписи и старым, и новым ключом во время переходного
периода). Подписывайте новые релизы новым ключом (`BCM_SIGNING_KEY`).
Когда все ноды обновятся на релиз с новым ключом — старый можно удалить.
